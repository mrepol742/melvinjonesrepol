اختيار مساعد برمجي بالذكاء الاصطناعي أصبح بطريقة ما أصعب من اختيار JavaScript framework. كل أسبوع هناك benchmark جديد، ولوحة ترتيب جديدة، ومنشور فيروسي جديد يدّعي أن نموذجًا ما سحق المنافسة بالكامل. ثم تستخدمها فعلًا في عمل حقيقي وتدرك أن الـ benchmarks لا تشرح لماذا يفهم مساعد سياقك المعماري بشكل خاطئ بينما يتمكن آخر بطريقة ما من إصلاح ثلاثة أخطاء غير مرتبطة في محادثة واحدة. بعد عدة أسابيع من التنقل بين Codex وClaude وGemini في مشاريع حقيقية بدلًا من أمثلة سطحية، توقفت عن السؤال عن "الأذكى". أصبح السؤال الأفضل: أيها أثق به في أجزاء مختلفة من سير عملي؟

الجواب لم يكن واضحًا كما توقعت، والمفاجأة أنه لا يوجد فائز واحد.

## التجربة

بدلًا من مقارنة coding benchmarks أو مطالبة كل نموذج بعكس binary tree للمرة الألف، منحت المساعدين الثلاثة المسؤوليات نفسها.

لم يكن المشروع مهيبًا بشكل خاص.

تطبيق full-stack فيه مصادقة وREST APIs وترحيلات قاعدة بيانات وDocker configuration وخطوط CI، ومع ما يكفي من الأخطاء لجعل الأمور ممتعة.

لم يكن الهدف معرفة من يكتب أكبر قدر من الكود.

بل معرفة من يمكنه مساعدتي في إنهاء المشروع.

## Codex يشبه البرمجة الثنائية مع مهندس

Codex فاجأني أكثر من غيره.

بدلًا من شرح ما تفعله كل سطر من الكود باستمرار، يميل إلى التركيز على حل المشكلة مباشرة. يبدو مرتاحًا في التنقل داخل codebase موجود، وتعديل ملفات متعددة، وفهم كيف تؤثر التغييرات في بقية المشروع.

الذي لفت انتباهي لم يكن الذكاء الخام.

بل الثقة أثناء العمل داخل مستودعات حقيقية.

عندما طلبت منه إعادة هيكلة authentication middleware، وتحديث unit tests ذات الصلة، وضبط routing logic، نادرًا ما تعامل معها كطلبات منفصلة. فهم أنها تنتمي معًا.

وهذا جعله يبدو أقل كمساعد وأكثر كمهندس آخر يعمل بجانبي.

### أين يتألق Codex

- إعادة هيكلة على مستوى المستودع
- Understanding project structure
- Multi-file code changes
- Software engineering workflows
- Writing production-ready implementations

### Where it struggles

Occasionally it assumes too much.

If your project has unusual architecture or undocumented conventions, you'll probably need to steer it back on course before it confidently heads in the wrong direction.

## Claude Thinks Before It Types

Claude has a very different personality.

It feels slower, but intentionally so.

Instead of immediately generating code, it often explains trade-offs first, identifies hidden edge cases, and points out implementation details I hadn't considered.

Sometimes that feels unnecessary.

Other times it saves an afternoon of debugging.

One example involved database transactions.

Rather than simply generating a migration script, Claude questioned whether the operation should even happen inside a single transaction because of potential locking issues.

That wasn't the answer I expected.

It was the answer I needed.

### Where Claude shines

- Large code reviews
- Debugging complicated systems
- Architecture discussions
- Explaining complex implementations
- Identifying hidden edge cases

### Where it struggles

Claude occasionally becomes overly cautious.

If you're looking for a quick implementation, it sometimes spends more time discussing possible approaches than actually writing the solution.

## Gemini Feels Like Having Google Built Into Your IDE

Gemini's biggest advantage isn't necessarily coding.

It's context.

Working inside Google's ecosystem gives it access to documentation, cloud services, APIs, and development resources in a way that feels incredibly natural.

When working with Firebase, Google Cloud, Android development, or modern web APIs, Gemini often suggested implementation details before I even searched for documentation myself.

It reduced the number of browser tabs I needed open.

That's an underrated productivity boost.

### Where Gemini shines

- Google Cloud projects
- Firebase applications
- Android development
- API documentation
- Research-heavy development

### Where it struggles

On larger repositories, it occasionally loses track of architectural decisions made earlier in the conversation.

It's excellent at solving individual problems but sometimes requires reminders about the bigger picture.

## The Real Difference Isn't Intelligence

One thing became obvious after several weeks.

None of these models consistently wrote dramatically better code than the others.

The difference was how they approached software engineering.

Codex wants to build.

Claude wants to reason.

Gemini wants to connect information.

Those personalities influence every interaction.

If you're debugging a distributed system, Claude often feels like the strongest partner.

If you're implementing features across multiple files, Codex usually moves faster.

If you're integrating cloud services or unfamiliar APIs, Gemini reduces research time significantly.

## Where Each Model Fits Best

| Situation                        | My First Choice |
| -------------------------------- | --------------- |
| Large codebase refactoring       | Codex           |
| Architecture planning            | Claude          |
| Debugging complex issues         | Claude          |
| Building new product features    | Codex           |
| Google Cloud development         | Gemini          |
| Firebase projects                | Gemini          |
| Learning unfamiliar technologies | Gemini          |
| Code review before deployment    | Claude          |

The funny part is that I rarely stick with just one anymore.

## The Workflow That Actually Worked

Eventually I stopped trying to crown a winner.

Instead, I gave each assistant a job.

Claude reviewed architecture.

Codex implemented features.

Gemini filled knowledge gaps and helped with documentation.

It sounds excessive until you realize experienced engineering teams already divide responsibilities among specialists.

We're starting to do something similar with AI.

Different tools.

Different strengths.

Same project.

## Benchmarks Don't Ship Products

Leaderboards are fun until deadlines appear.

When you're trying to finish a feature before the weekend, nobody cares if a model scored two percentage points higher on a benchmark published six months ago.

You care whether it understands your codebase, catches subtle bugs, respects your project structure, and saves enough time that you can finally close the laptop before midnight.

After using all three extensively, I've stopped asking which AI is the best.

I ask which one I'd invite to solve the problem sitting in front of me.

More often than not, the answer depends on the problem, not the model.
